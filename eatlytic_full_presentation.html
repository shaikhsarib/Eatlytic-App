Eatlytic — Complete Project Blueprint

A simple, comprehensive explanation of our system architecture, the 8 core modules, scan flows, and roadmap.

01. What is Eatlytic? (The Core Concept)

Part 1

Eatlytic is a Personal Nutrition Intelligence Layer (PNIL). Instead of just adding up calories (like general fitness trackers), Eatlytic decodes the food label to reveal its physical truth and safety profile.

In Simple Words: It validates product claims using physical laws (thermodynamics), detects processed food additives, and checks if food is safe for a user's health condition (like diabetes) or genetic profile (lactose intolerance, sodium sensitivity).

Key Core Features:

  • Claim Verification: Catches brands lying about calories or ingredients at run-time.
  • Offline First Design: Runs deterministic rules locally before triggering cloud AI models, saving significant server costs.
  • CGM & DNA Integrations: Correlates scanned items with continuous glucose monitor spikes and genetic markers.

02. Codebase Walkthrough — The 8 Core Modules

Part 2

The codebase is split into specific modules. Here is exactly what they do in plain English:

brain.py (The Local Brain)

Takes the parsed nutrients and runs offline medical logic. It assigns a safety rating (Safe/Limit/Avoid) based on user conditions (e.g. diabetes) without needing expensive LLMs.

fake_detector.py (The Lie Detector)

Runs Atwater Calorie calculations: (protein*4) + (carbs*4) + (fat*9) to catch calorie fraud. It also scans ingredients lists for hidden sugars to catch fake "No Added Sugar" claims.

scan_orchestrator.py (The Traffic Cop)

Manages the entire scan pipeline: handles OCR extraction fallbacks, routes matches, triggers RAG searches, runs the clinical engine, and stores the results.

personalized_medicine.py (The DNA Layer)

Checks user genetic SNPs for diabetes predisposition (TCF7L2), sodium-sensitive hypertension (AGT), and lactose intolerance (LCT) to override safety scores.

cgm.py (The Biosensor Engine)

Manages continuous glucose monitor feeds, calculates Time-in-Range (70-140 mg/dL), and links food scans with actual glucose spikes 2 hours post-meal.

additive_db.py (The Additive Lookup)

Loads additives.json to perform rapid offline lookups on E-numbers and INS codes (like MSG or Tartrazine) for safety status and FSSAI rules.

dietitian.py (The B2B Portal)

A clinical dashboard allowing professional dietitians to track their patients' scans, CGM spikes, compliance reports, and send medical interventions.

mcp.py (The AI Server Connection)

Integrates Model Context Protocol (MCP) to let external AI systems query Eatlytic's database and metabolic compatibility scoring dynamically.

03. Step-by-Step Scan Lifecycle

Part 3

How a scan moves through Eatlytic when a user photographs a label package:

1. OCR Text Extracted (OCR engine reads the label image) │ ▼ 2. Local Database Match (Bypasses LLM / Web queries if product barcode matches database) │ [No Match] ▼ 3. Query Formulation (Extracts top 4 tokens + brand name to form a search key) │ ▼ 4. Cache Verification (Checks L1 memory dictionary -> L2 SQLite/Supabase 'research_cache') │ [Cache Miss] ▼ 5. Web Search (RAG) (Performs a fast DuckDuckGo context search for safety advisories) │ ▼ 6. Clinical & Physics Audits (Atwater calorie calculations + Persona warning flags) │ ▼ 7. Genomic Check (Applies DNA lactose/diabetic/sodium overrides) │ ▼ 8. Save Scan & Cache result (Returns a detailed report card to the user)

04. RAG Engine & Live Context Retrieval

Part 4

How the real-time web-augmented RAG (Retrieval-Augmented Generation) pipeline operates to pull live product updates and safety alerts:

How it works: If a scanned product misses our local database moat, the orchestrator formulates a search query from OCR keywords and triggers a background DuckDuckGo search (with a 3-second timeout). Results are cached in two tiers (L1 memory dict + L2 SQLite/Supabase DB table research_cache) before being injected straight into the LLM system prompt.

Subsystem Architecture Map:

Vulnerabilities & Issues to Address:

  • Scraper Fragility: DuckDuckGo rate-limits scrapers. When it blocks us, RAG fails silently, causing the LLM to score without live data. We must migrate to a local vector database of FSSAI regulations.
  • Generic Search Terms: Grabbing the first 4 words of a label layout often queries generic words like "Nutrition Facts Serving Size". We need to refactor query formulation to search based on the *detected ingredients*.

05. The Challenges & Dev Roadmap

Part 5

What the development team is focusing on next to launch Eatlytic successfully:

1. Core Technical Goals:

  • Scraping FSSAI Permitted Additives: Grow the database from 76 additives and 27 ingredients to 500+ items using public regulatory datasheets.
  • Replacing DDG Scraping: Web scraping is fragile and gets blocked. We will implement local vector databases loaded with FSSAI rules and food manuals.
  • Harden DB Connections: Verify WAL configurations to support multiple concurrent users on SQLite.
